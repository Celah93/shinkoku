# クレジットカード明細から弥生会計23振替伝票を生成する機能の実装前調査報告書

- 調査日: 2026-07-17
- 対象ワークスペース: `C:\Users\User\Documents\GPT_CODEX_Projects\shinkoku`
- 対象バージョン: `0.6.12`
- 調査対象: クレジットカード明細CSVの読込み、仕訳候補の作成、帳簿登録、
  および弥生会計23向け振替伝票テキストの生成
- 関連資料:
  - `docs/yayoi23-import-investigation-2026-07-17.md`
  - `docs/yayoi-export-invoice-design-preinvestigation-2026-07-17.md`

## 1. 結論

クレジットカード明細を読み取り、弥生会計23へ振替伝票として取り込める`.txt`を
生成する機能は実装可能である。ただし、現行ツリーには、次の処理が実装されていない。

1. カード会社ごとに異なる明細CSVを、カード取引として正規化する処理。
2. カードの利用時と口座からの引落時とを分けて、適切な仕訳へ変換する処理。
3. 仕訳を弥生会計の25列形式へ変換する専用ライター。
4. 25列のデータを、CP932、CRLF、ヘッダーなしの`.txt`として保存する処理。
5. 同日、同額、同科目の正当なカード利用を、誤って完全重複として停止しないための
   取込元行識別処理。

弥生会計上の「振替伝票」として取り込む場合には、通常仕訳用の
`識別フラグ=2000`および`タイプ=0`ではなく、単純な1行伝票では
`識別フラグ=2111`および`タイプ=3`を組み合わせる必要がある。
費用の分割、税率の分割、または家事按分によって複数行になる伝票では、
`2110`、`2100`、`2101`を順番に使用し、タイプには`3`を使用する必要がある。

現行の帳簿機能を活用するため、推奨する実装経路は次のとおりである。

```text
カード明細CSV
  -> カード明細専用の取込処理
  -> 正規化した仕訳候補
  -> ユーザーによる事業性、勘定科目、税区分の確認
  -> JournalEntryへの変換
  -> journal-batch-addによる帳簿登録
  -> 登録済みjournal_idを指定した弥生振替伝票出力
  -> 弥生会計23用.txt
```

明細CSVから弥生ファイルへ直接変換すると、SHINKOKUの帳簿、重複確認、監査ログ、
および決算処理と出力内容とが分離する。そのため、帳簿登録後に弥生ファイルを出力する
DB先行の経路を推奨する。

## 2. 調査範囲と用語

### 2.1 本調査で扱う範囲

本調査では、次の範囲を対象にする。

- 事業用クレジットカードの通常購入および返金。
- 個人用クレジットカードで支払った事業経費。
- カード利用時の未払計上。
- 口座引落時の未払金の消込。
- 単一仕訳および複合仕訳を、弥生会計の振替伝票として出力する処理。
- 入力ファイルの重複および入力行の重複を検出する処理。

### 2.2 初回実装の対象外とすることが妥当な取引

次の取引は、明細の表示額だけでは取得価額、元本、利息、手数料、または為替差額を
正確に判定できない場合があるため、初回実装では自動確定の対象外にすることが妥当である。

- 分割払いおよびリボルビング払い。
- キャッシングおよびカードローン。
- 外貨建て取引および海外利用手数料。
- チャージバック、取消し、および一部返金で、元取引との対応を確認できない取引。
- 複数月をまたぐ支払変更および繰上返済。

これらの行は読み飛ばさず、`needs_review`として返し、利用者による確認後にだけ
仕訳へ変換する必要がある。

### 2.3 本報告書でいう「振替伝票」

本報告書でいう振替伝票は、弥生会計へインポートした後に、画面上のタイプが
「振伝」になる伝票を指す。単に借方と貸方とを持つ仕訳を、一般的な意味で
「振替仕訳」と呼んでいるわけではない。

## 3. 弥生会計の正式仕様から確定できる要件

### 3.1 識別フラグとタイプとの組合せ

弥生会計の正式な仕訳インポート仕様では、識別フラグとタイプとは別の列である。

| 出力目的 | 識別フラグ | タイプ | SHINKOKUでの使用方針 |
|---|---:|---:|---|
| 伝票ではない通常仕訳 | `2000` | `0` | 通常仕訳として出力する場合だけ使用する |
| 1行の振替伝票 | `2111` | `3` | 借方1行、貸方1行のカード取引で使用する |
| 複数行振替伝票の先頭行 | `2110` | `3` | 複合仕訳の先頭で使用する |
| 複数行振替伝票の中間行 | `2100` | `3` | 複合仕訳の中間で使用する |
| 複数行振替伝票の最終行 | `2101` | `3` | 複合仕訳の末尾で使用する |

既存の実機調査では、`2111`と`0`とを組み合わせたため、弥生会計23が
「識別フラグと取引タイプが矛盾しています」として取込みを停止した。
この結果は、`docs/yayoi23-import-investigation-2026-07-17.md`に記録されている。

正式仕様の参照先は、次の資料である。

- [弥生会計: 仕訳データの項目と記述形式](https://support.yayoi-kk.co.jp/subcontents.html?grade_id=Blue&page_id=18545)
- [弥生会計: 帳簿で入力した仕訳を振替伝票に変換](https://support.yayoi-kk.co.jp/subcontents.html?grade_id=Pro1U&page_id=17776)

### 3.2 25列の推奨マッピング

単純なカード利用を1行の振替伝票として出力する場合には、次のマッピングが妥当である。

| 列 | 弥生項目 | 推奨値または取得元 |
|---|---|---|
| A | 識別フラグ | 単純仕訳では`2111`、複合仕訳では`2110`、`2100`、`2101` |
| B | 伝票No. | 初回実装では空欄にして、弥生側の付番設定に従う |
| C | 決算 | 通常取引では空欄にする |
| D | 取引日付 | カード利用仕訳では利用日、引落仕訳では引落日を使用する |
| E | 借方勘定科目 | `account_code`を弥生の勘定科目名へ変換する |
| F | 借方補助科目 | カードまたは口座を補助科目で管理する場合に設定する |
| G | 借方部門 | 初回実装では空欄にする |
| H | 借方税区分 | SHINKOKUの税区分およびインボイス判定から変換する |
| I | 借方金額 | 税込の整数円を設定する |
| J | 借方税金額 | 税抜経理で明示が必要な場合に設定する |
| K | 貸方勘定科目 | 事業用カードでは未払金、個人カードでは事業主借などを設定する |
| L | 貸方補助科目 | 事業用カードではカード名を設定できるようにする |
| M | 貸方部門 | 初回実装では空欄にする |
| N | 貸方税区分 | 未払金、事業主借、および普通預金では通常は`対象外`を使用する |
| O | 貸方金額 | 税込の整数円を設定する |
| P | 貸方税金額 | 通常は`0`または空欄とし、弥生実機で許容値を検証する |
| Q | 摘要 | 利用店名および取引内容を設定する |
| R | 番号 | 10文字以内の取引識別子を設定する場合に使用する |
| S | 期日 | 引落予定日を保持する設計にした場合にだけ設定する |
| T | タイプ | 振替伝票では`3`を設定する |
| U | 生成元 | 初回実装では空欄にする |
| V | 仕訳メモ | カードID、入力ファイル名、および入力行番号を設定する候補とする |
| W | 付箋1 | 初回実装では`0`を設定する |
| X | 付箋2 | 初回実装では`0`を設定する |
| Y | 調整 | 通常取引では`no`を設定する |

勘定科目名は、弥生会計側に登録されている名称と一致する必要がある。SHINKOKUの
科目コードから`MASTER_ACCOUNTS`の名称を取得するだけでは、利用者が弥生側で変更した
科目名または補助科目名へ対応できない。そのため、弥生用の科目名および補助科目名を
上書きできるマッピングを用意する必要がある。

### 3.3 複数行振替伝票

1件のカード利用を複数の費用、税率、または事業割合へ分割する場合には、
1行の`2111`では表現できない。1件の`JournalEntry`を1枚の振替伝票として扱い、
次の規則で出力する必要がある。

1. 最初の出力行では、識別フラグに`2110`を設定する。
2. 中間の出力行では、識別フラグに`2100`を設定する。
3. 最後の出力行では、識別フラグに`2101`を設定する。
4. 借方または貸方が存在しない出力行では、勘定科目を空欄にし、必須金額を`0`にする。
5. 伝票全体では、借方合計と貸方合計とを一致させる。
6. タイプには`3`を設定する。

借方だけを持つ行と貸方だけを持つ行との並べ方は、正式仕様上では許容されているが、
弥生会計23での表示順および空欄側の扱いは、実機の受入試験で確認する必要がある。

### 3.4 ファイル形式

初回実装では、次の形式へ固定することが妥当である。

- 拡張子は`.txt`にする。
- 区切り文字にはカンマを使用する。
- ヘッダー行は出力しない。
- 1行には必ず25列を出力する。
- 文字コードにはCP932を使用する。
- 改行コードにはCRLFを使用する。
- BOMは付与しない。
- CP932で表現できない文字がある場合には、`?`へ置換せず、対象の仕訳IDと項目名とを
  含むエラーとして停止する。

25列の項目順は弥生の正式仕様による。CP932、CRLF、および`.txt`の組合せは、
`docs/yayoi23-import-investigation-2026-07-17.md`に記録された弥生会計23の
実機取込結果による。

## 4. クレジットカード取引の会計処理

弥生会計の公式資料では、事業用カードの利用額は後日に決済されるため、利用時には
未払金を計上する。引落時には、未払金を普通預金で消し込む。

- [弥生会計: クレジットカードで支払ったとき](https://support.yayoi-kk.co.jp/subcontents.html?page_id=25642)
- [弥生会計: 経費を個人用の現預金やカードで支払ったとき](https://support.yayoi-kk.co.jp/business/faq_Subcontents.html?page_id=694)

SHINKOKUの既存マスターには、普通預金`1002`、未払金`2030`、事業主借`3010`、
および事業主貸`1200`が存在する。主要な仕訳は次のとおりである。

| 取引 | 借方 | 貸方 | 実装上の扱い |
|---|---|---|---|
| 事業用カードで事業経費を支払った | 費用または資産 | 未払金`2030` | カード利用日に仕訳を作成する |
| 事業用カードの代金が事業口座から引き落とされた | 未払金`2030` | 普通預金`1002` | 銀行明細から作成することを推奨する |
| 個人用カードで事業経費を支払った | 費用または資産 | 事業主借`3010` | 原則として、事業帳簿では引落仕訳を作成しない |
| 事業用カードで個人的な支出をした | 事業主貸`1200` | 未払金`2030` | 事業用カードの未払残高へ含める |
| 事業用カードの購入が返金された | 未払金`2030` | 元の費用または資産 | 金額を負数のまま登録せず、貸借を反転する |

現在の`JournalLine.amount`は正の整数だけを受け付ける。そのため、明細CSVに返金が
負数として記録されている場合には、絶対値を使用し、借方と貸方とを反転する必要がある。

カード明細から引落仕訳も生成し、銀行明細から同じ引落仕訳も生成すると、二重計上に
なる。初回実装では、カード明細からは利用仕訳だけを生成し、引落仕訳は銀行明細から
生成する方針を推奨する。カード明細に確定引落額が含まれる場合でも、初期値では
引落仕訳を生成しない必要がある。

## 5. 現行実装の状態

### 5.1 設定

`ShinkokuConfig`には`credit_card_statements_dir`があるが、値はディレクトリのパスだけである。
カード名、カードの所有区分、未払金科目、引落口座、締日、支払日、CSV列マッピング、
および弥生の補助科目は保持しない。

出典: `src/shinkoku/config.py:127-130`、`shinkoku.config.example.yaml:21-24`

### 5.2 CSV読込み

`import_csv`には、次の機能がある。

- UTF-8、UTF-8 BOM、およびCP932の判定。
- 日付、摘要、および金額の列の簡易検出。
- `YYYY-MM-DD`および`YYYY/MM/DD`の日付の正規化。
- 通貨記号、3桁区切り、全角数字、および複数の負数表現の金額解析。
- 元の列値を`original_data`へ保持する処理。
- 入力ファイルのSHA-256ハッシュの計算。

一方で、次の機能はない。

- カード会社別のCSV仕様を表すプロファイル。
- 年を含まない利用日の補完。
- 利用日、売上日、請求日、および支払日の区別。
- 利用額、返金額、手数料、および支払額の区別。
- カードID、支払区分、通貨、利用者名、および取引IDの保持。
- 事業取引、私用取引、および確認待ち取引の区別。
- 勘定科目および税区分の決定。

出典: `src/shinkoku/tools/import_data.py:16-221`

テスト用の`tests/fixtures/csv/credit_card_simple.csv`は、利用日、利用店名、および
利用金額だけを持つ簡単な5行のCSVである。このフィクスチャは、カード取引専用の
変換処理または弥生出力を検証していない。

### 5.3 仕訳登録

`JournalEntry`は、日付、摘要、取引先、複数の借貸明細、取込元、および取込元ファイルを
保持できる。`journal-batch-add`は、複数の仕訳を一括登録できる。そのため、確認済みの
カード取引をSHINKOKUの帳簿へ登録する基盤は存在する。

一方で、`JournalEntry`には、カードID、入力行番号、入力側の取引ID、弥生の補助科目、
弥生の伝票タイプ、およびエクスポート済み状態がない。

出典: `src/shinkoku/models.py:11-30`、`src/shinkoku/tools/ledger.py:76-231`

### 5.4 弥生出力

現行の`ledger`コマンドが受け付ける出力形式は、`json`または一般的な`csv`だけである。
一般CSVは標準出力へ出力され、弥生の25列、識別フラグ、タイプ、CP932、ファイル拡張子、
および出力履歴を扱わない。

出典: `src/shinkoku/cli/ledger.py:122-273, 866-867`

### 5.5 重複検出

現行の`compute_journal_hash`は、日付、貸借、勘定科目コード、および金額からハッシュを
作成し、摘要、取引先、取込元ファイル、および入力行番号を含めない。

そのため、同じ日に同じ店で同額を2回支払った場合だけでなく、異なる店であっても
同じ費用科目と同じ金額とを使用すると、完全一致として登録を停止する可能性がある。
カード明細の一括登録では、この状況が現金レシートよりも発生しやすい。

ファイル単位の`import_sources`は存在するが、入力行と仕訳との対応を保持しない。
同じファイルの再取込みは検出できても、前月と当月とに重複して含まれた利用行、
再ダウンロードによってファイル内容が変わった明細、または同一ファイル内の正当な
同日同額取引を安全に区別できない。

出典: `src/shinkoku/hashing.py:10-24`、`src/shinkoku/duplicate_detection.py:18-97`、
`src/shinkoku/schema.sql:400, 408`

## 6. 推奨するカード明細モデル

### 6.1 カードプロファイル

カード会社ごとのCSV差異をソースコード内の条件分岐へ直接埋め込まず、入力プロファイルで
表現することを推奨する。モデル名は、既存の命名規約に合わせて
`CreditCardStatementProfile`または`CreditCardImportInput`とすることが妥当である。

最低限、次の項目が必要である。

| 項目案 | 内容 |
|---|---|
| `card_id` | SHINKOKU内でカードを識別する安定したID |
| `card_name` | 画面表示、摘要、および弥生補助科目に使用する名称 |
| `ownership` | `business`または`personal` |
| `statement_year` | 年を含まない日付を補完するための対象年 |
| `date_column` | 利用日を保持する列名または列番号 |
| `description_column` | 利用店名または摘要を保持する列名または列番号 |
| `amount_column` | 利用金額を保持する列名または列番号 |
| `transaction_id_column` | カード会社の取引IDを保持する列。存在しない場合は`null` |
| `payment_type_column` | 一括、分割、リボなどを保持する列。存在しない場合は`null` |
| `amount_sign` | 正数が購入を表すか、返金を表すかという規則 |
| `liability_account_code` | 事業用カードで使用する未払金科目。初期値は`2030` |
| `owner_account_code` | 個人カードで使用する事業主借科目。初期値は`3010` |
| `payment_account_code` | 引落仕訳で使用する預金科目。初期候補は`1002` |
| `yayoi_liability_subaccount` | 弥生側でカードを区別する補助科目名 |

### 6.2 正規化した取引候補

専用インポーターは、汎用`import_csv`の戻り値をそのまま仕訳へ変換せず、
`CreditCardTransactionCandidate`のような中間モデルを返すことが妥当である。

最低限、次の項目が必要である。

| 項目案 | 内容 |
|---|---|
| `source_file` | 入力ファイルのパス |
| `source_file_hash` | 入力ファイルのSHA-256ハッシュ |
| `source_row_number` | ヘッダーを含む入力行番号 |
| `source_transaction_id` | カード会社が付与した取引ID |
| `source_row_key` | ファイル、カード、および行を識別するキー |
| `card_id` | カードプロファイルのID |
| `usage_date` | 正規化した利用日 |
| `merchant` | 利用店名 |
| `amount` | 絶対値の整数円 |
| `transaction_kind` | `purchase`、`refund`、`fee`、または`needs_review` |
| `business_use` | `business`、`personal`、または`unknown` |
| `expense_account_code` | 確認後に設定する費用または資産の科目コード |
| `tax_category` | 確認後に設定するSHINKOKUの税区分 |
| `original_data` | 入力行の元の列値 |
| `warnings` | 日付補完、金額符号、支払区分などの警告 |

`expense_account_code`、`tax_category`、および`business_use`が未確定の候補は、
弥生ファイルへ出力してはならない。

## 7. 推奨するCLIと処理境界

### 7.1 カード明細の取込み

汎用`import csv`との後方互換性を保ち、カード固有の処理を明示するため、次のような
専用サブコマンドを追加することが妥当である。

```bash
uv run shinkoku import credit-card \
  --file-path statement.csv \
  --input card-profile.json
```

このコマンドは、DBへ自動登録せず、正規化した候補、除外行、および確認待ち行を
JSONで返す必要がある。

### 7.2 帳簿登録

ユーザーが事業性、勘定科目、税区分、および返金の対応を確認した後に、既存の
`journal-batch-add`を使用する。

```bash
uv run shinkoku ledger journal-batch-add \
  --db-path shinkoku.db \
  --fiscal-year 2025 \
  --input confirmed-card-journals.json
```

登録結果が返す`journal_ids`を、次の弥生出力へ渡す必要がある。

### 7.3 弥生振替伝票の出力

弥生固有の処理を一般CSV出力から分離するため、新しい`yayoi`コマンド群を追加することを
推奨する。

```bash
uv run shinkoku yayoi export \
  --db-path shinkoku.db \
  --input yayoi-export.json \
  --output-path output/yayoi-credit-card-2025-01.txt
```

`yayoi-export.json`では、少なくとも次の値を受け取る必要がある。

```json
{
  "journal_ids": [101, 102, 103],
  "voucher_type": "transfer",
  "strict": true
}
```

CP932、CRLF、ヘッダーなし、および25列は、弥生会計23向けの固定仕様とし、
初回実装では利用者が変更できないようにすることが妥当である。

## 9. 重複検出と取込元行識別

### 9.1 現行ハッシュをそのまま使用する場合の問題

次の2件は、異なる店で行った正当な取引であっても、現行ハッシュでは完全一致になる。

```text
2025-01-10 A店 消耗品費 1,000円 / 未払金 1,000円
2025-01-10 B店 消耗品費 1,000円 / 未払金 1,000円
```

摘要をハッシュへ入れるだけでは、店名表記の変更、全角と半角との差、カード会社側の
表記揺れ、および再ダウンロード時の表記変更に弱い。一方で、摘要を含めなければ、
正当な同日同額取引を区別できない。

### 9.2 推奨する識別方針

内容ハッシュと取込元識別子とは、別の目的として保持する必要がある。

1. 入力ファイルの再処理は、既存の`file_hash`で検出する。
2. 入力行の再処理は、カード会社の取引IDがある場合には、そのIDと`card_id`とで検出する。
3. 取引IDがない場合には、`file_hash`と`source_row_number`とで同一ファイル内の行を識別する。
4. 異なるファイル間で同一の可能性がある行は、利用日、金額、店名、カードID、および
   支払区分から類似候補として提示し、自動削除しない。
5. 同日、同額、同科目であっても、異なる安定した取込元行キーを持つ場合には、
   完全一致として停止しない。

実装方法としては、`journals`へ`source_record_key`を追加する方法、または
`import_source_rows`のような専用テーブルで入力行と`journal_id`とを関連付ける方法がある。
複数の仕訳が1つの入力行から作られる可能性、ならびに将来の取消しおよび再分類を考えると、
専用テーブルを採用する方が拡張しやすい。

この設計を決めずに一括取込を実装すると、利用者が正当な取引ごとに`--force`を指定する
運用になり、カード明細の自動化としては不十分になる。

## 10. 税区分とインボイス制度との関係

カード明細には、支払先、利用日、および総額だけが記録される場合が多く、レシートまたは
適格請求書の情報が含まれない。カード利用明細だけでは、適格請求書発行事業者の登録番号、
取引日時点の有効性、税率別金額、請求書区分、および仕入税額控除割合を確定できない。

したがって、次の規則が必要である。

- カード明細だけを根拠として、税区分の末尾へ`適格`を自動付与しない。
- レシートまたは請求書と照合できた場合にだけ、既存のインボイス判定結果を使用する。
- `tax_category`が未設定の場合には、弥生出力で黙って課税仕入へ補完しない。
- `taxable_10`、`taxable_8`、`taxable_8_reduced`、`non_taxable`、`exempt`、および
  `out_of_scope`から弥生税区分への変換表を、専用の定数または関数へ集約する。
- 返金では、元取引と対応する仕入返還の税区分を使用する必要があるため、元取引が
  不明な返金を自動確定しない。

標準税率10%の税込課税仕入では`課対仕入込10%`が候補になる。軽減税率8%の税込課税仕入では
`課対仕入込軽減8%`が候補になる。請求書区分および控除割合のサフィックスは、
`docs/yayoi-export-invoice-design-preinvestigation-2026-07-17.md`の設計課題と共通である。

正式な税区分名称の参照先は、次の資料である。

- [弥生会計: 課税方式別税区分・税計算区分一覧](https://support.yayoi-kk.co.jp/subcontents.html?page_id=18111)

## 11. エラー処理

専用インポーターおよび弥生ライターは、次の状況で処理を停止するか、該当行を
`needs_review`へ移す必要がある。

- 利用日、利用店名、または金額の列を一意に特定できない。
- 年を含まない日付に対して、対象年を決定できない。
- 金額の符号規則を決定できない。
- 金額が0円である。
- 支払区分が分割、リボ、キャッシング、または不明である。
- 事業取引か私用取引かを決定できない。
- 勘定科目または税区分が未設定である。
- 借方合計と貸方合計とが一致しない。
- 弥生の勘定科目名または税区分名が上限文字数を超える。
- 摘要、メモ、または科目名にCP932で表現できない文字が含まれる。
- 1行が25列にならない。
- 複数行伝票の識別フラグの並びが不正である。
- 出力対象の`journal_id`が存在しない、年度が異なる、または重複して指定されている。

弥生会計は、以前に取り込んだ仕訳を再度取り込んでも、警告せずに重複登録する。
そのため、SHINKOKU側では、出力ファイルのハッシュ、対象`journal_ids`、出力日時、
および出力先を記録することが望ましい。

- [弥生会計: 仕訳データを取り込む方法](https://support.yayoi-kk.co.jp/faq_Subcontents.html?page_id=18043)
- [弥生会計: インポートのエラーログファイル](https://support.yayoi-kk.co.jp/subcontents.html?grade_id=Std&page_id=18438)

## 12. 必要なテスト

### 12.1 カード明細の単体テスト

1. UTF-8、UTF-8 BOM、およびCP932の各明細を読み取れる。
2. 列名をプロファイルで明示した場合には、自動検出よりも明示値を優先する。
3. 年を含まない日付を、対象年と請求期間とに基づいて正規化する。
4. 購入と返金との符号を、カードプロファイルに基づいて正規化する。
5. 0円、分割、リボ、キャッシング、および不明行を`needs_review`へ移す。
6. 同じファイルを再処理した場合には、ファイル重複として検出する。
7. 同一ファイル内の同日同額取引を、異なる行として保持する。
8. カード会社の取引IDが同じ行を、入力行重複として検出する。

### 12.2 仕訳変換の単体テスト

1. 事業用カードの購入を、費用または資産の借方と未払金の貸方とへ変換する。
2. 個人カードの事業経費を、費用または資産の借方と事業主借の貸方とへ変換する。
3. 事業用カードの私用取引を、事業主貸の借方と未払金の貸方とへ変換する。
4. 返金では、金額を正の整数にし、元取引の貸借を反転する。
5. 引落仕訳を生成しない既定動作を検証する。
6. 明示的に有効化した場合だけ、未払金と普通預金との引落仕訳を生成する。
7. 同日同額同科目で、異なる取込元行キーを持つ2件を登録できる。

### 12.3 弥生出力の単体テスト

1. 単純仕訳を`2111`およびタイプ`3`で出力する。
2. 複合仕訳を`2110`、`2100`、`2101`およびタイプ`3`で出力する。
3. 各出力行が必ず25列になる。
4. ヘッダーを出力しない。
5. 最終バイト列がCP932であり、BOMを持たず、改行がCRLFになる。
6. CP932で表現できない文字を、置換せずにエラーにする。
7. カンマ、二重引用符、および改行を含む摘要を、安全にCSVエスケープするか、
   改行を拒否するという決定済みの方針に従う。
8. 弥生の各項目の文字数制限を検証する。
9. 借方および貸方の税区分を必須として検証する。
10. 未払金、事業主借、普通預金、および事業主貸の税区分を`対象外`として出力する。
11. 返金に仕入返還の税区分を設定する。
12. 出力対象に年度外の仕訳が含まれる場合には停止する。

### 12.4 CLIおよび結合テスト

1. `import credit-card`がJSONを標準出力へ返す。
2. `journal-batch-add`の登録結果から`journal_ids`を取得できる。
3. `yayoi export`が指定した出力先へ`.txt`を作成する。
4. 生成したファイルを再読込みし、列数、金額、および伝票グループを検証できる。
5. 同じ`journal_ids`で再出力する場合には、既出力であることを警告する。

### 12.5 弥生会計23での実機受入試験

自動テストだけでは、弥生会計23の画面上で「振伝」として表示されること、補助科目の
マッチング、税区分の補完、および複数行伝票の表示順を確認できない。実機では、
必ずバックアップを作成した検証用事業所データへ、次のサンプルを取り込む必要がある。

1. 事業用カードによる10%課税仕入。
2. 軽減税率8%を含む仕入。
3. 個人カードによる事業経費。
4. 事業用カードによる私用取引。
5. 返金。
6. 借方を2科目へ分割した複数行伝票。
7. 税率を10%と軽減8%とへ分割した複数行伝票。
8. 未払金を普通預金で消し込む引落仕訳。

取込み後には、タイプが「振伝」であること、借貸が一致すること、税区分が意図どおりで
あること、請求書区分および仕入税額控除に警告が出ないこと、ならびに再インポートを
行っていないことを確認する必要がある。

## 14. 実装開始前に確定する必要がある事項

次の項目は、実装者が暗黙に決めず、仕様として明示する必要がある。

1. 初回対象にするカード会社および実際のCSVサンプル。
2. 事業用カードと個人用カードとの両方を初回から扱うかという範囲。
3. 1件の利用を複数科目または複数税率へ分割する処理を初回から含めるかという範囲。
4. 引落仕訳を銀行明細だけから作るかという方針。
5. 弥生側でカードごとの補助科目を使用するかという方針。
6. 入力行と仕訳との対応を、`journals`のカラムで保持するか、専用テーブルで保持するか
   という設計。
7. 弥生出力済み履歴をDBへ保存するかという設計。
8. 税区分が未確定の行を全面停止するか、`対象外`で仮出力する選択肢を設けるかという方針。
9. 弥生会計23のエディション、更新状態、およびインボイス対応状況。

本調査としては、次の初期値を推奨する。

- 事業用カードと個人用カードとは、カードプロファイルで明示的に区別する。
- 単純仕訳と複数行仕訳との両方を、振替伝票として出力する。
- カード明細からは利用仕訳だけを作り、引落仕訳は銀行明細から作る。
- カードごとの補助科目を設定できるようにするが、未設定も許可する。
- 税区分、事業性、または勘定科目が未確定の行は、弥生出力を停止する。
- 取込元行の識別には専用テーブルを使用し、内容ハッシュとは分離する。
- 出力済みの`journal_ids`と出力ファイルハッシュとを保存する。
