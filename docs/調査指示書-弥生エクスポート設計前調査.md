# 調査指示書: 弥生会計エクスポート・インボイス対応の設計前調査

- 作成日: 2026-07-17
- 種別: 事前調査（実装は行わない）
- 前提資料: `yayoi23-import-investigation-2026-07-17.md`（弥生会計23 仕訳インポート調査報告書）
- 対象ワークスペース: `shinkoku-main` 現行ツリー

## 1. 目的

弥生会計向け仕訳エクスポートと、適格請求書（インボイス）対応の恒久実装を設計する前に、
現行コードベースの事実関係を確定させる。

本調査の中心論点は次の1点である。

> **納税者自身の課税方式（免税 / 2割特例 / 簡易課税 / 本則課税）と
> 適格請求書発行事業者としての登録状況が、shinkoku のどこに・どの形式で保持され、
> どの処理がそれに依存しているか。**

課税方式によって、仕入側の適格判定が税額計算に影響するかどうかが変わる。
2割特例・簡易課税では仕入税額控除は売上税額から機械的に算出されるため、
レシートの登録番号照合は税額計算上不要である。本則課税でのみ必須となる。
この事実関係を確定させないまま実装範囲を決めると、
不要な照合パイプラインを優先実装するか、逆に本則課税で必須の判定を欠くかの
いずれかのリスクがある。

## 2. 実装しない理由（Why-not: なぜ調査のみか）

- 弥生税区分のサフィックス表記（`適格` / `区分80%` 等）の正確な文字列は、
  弥生公式仕様の一次確認が完了していない。誤った文字列で実装すると
  一字違いで補完動作が変わるため、仕様確定前の実装は行わない。
- 国税庁公表サイトの照合方式（Web-API か全件データか）は、利用規約・
  申請要件の確認が未了であり、方式決定前に照合コードを書くべきでない。
- 過去の Fix において、事前調査を省略した仕様が誤りを含んでいた事例がある。
  本件は税区分文字列・制度日付・課税方式分岐が絡むため、同様の手順を踏む。

外部一次資料（弥生仕様ページ・国税庁公表サイト規約・令和8年度税制改正資料）の
確認は Claude / ユーザー側の分担とする。本指示書の範囲はリポジトリ内調査に限る。

## 3. 調査項目

### A. 納税者属性の保持状況（最重要）

1. `shinkoku.config.yaml`、DB スキーマ、`/setup` スキル、`/assess` スキル、
   `.shinkoku/progress/02-assess.md` 引継書のそれぞれについて、
   以下の属性がどこに・どの名前で・どの型で保持されるかを確認する。
   - 課税事業者か免税事業者か
   - 申告方法（2割特例 / 簡易課税 / 本則課税）
   - 適格請求書発行事業者の登録有無・自身の登録番号
   - 簡易課税の事業区分（第1種〜第6種）
2. 上記が**永続化されているか**（config / DB）、それとも
   **セッション内の引継書・入力 JSON にしか存在しないか**を区別して報告する。
   `ConsumptionTaxInput.method` のように計算時に都度渡す設計であれば、
   「システムとして納税者の課税方式を知らない」状態である。その場合、
   仕訳登録時・エクスポート時に課税方式を参照できないことを意味するため、
   どの層で保持すべきかの選択肢（config 追加 / fiscal_years テーブル拡張 等）を
   実装せずに列挙する。

### B. 消費税計算の依存関係

1. `calc-consumption` の本則課税ルートが `taxable_purchases_10` / `taxable_purchases_8`
   をどこから受け取るかを追跡する。帳簿（`journal_lines.tax_category`）から
   自動集計する経路が存在するか、ユーザーが手動集計して JSON に渡す前提かを確認する。
2. 本則課税の仕入集計において、**免税事業者からの仕入れ（経過措置80%控除）を
   区別する仕組みが現状あるか**を確認する。ないことが予想されるが、
   「ない」ことを確認して報告すること（この欠落は弥生エクスポートとは独立に、
   本則課税ユーザーの税額計算そのものの正確性に関わる）。

### C. 仕訳データモデルと税区分の値域

1. `journal_lines.tax_category` と `tax_rate` の値域（実際に書き込まれる値の一覧）を、
   スキーマ定義・書き込み箇所・account-master の tax_category 初期値から確定する。
2. 請求書区分・登録番号・控除割合に相当するカラムや項目が、
   `journals` / `journal_lines` / 中間モデルのどこにも存在しないことを確認する。
3. 既存 DB に対して後方互換を保ったままカラム追加する場合の制約
   （マイグレーション機構の有無、`content_hash` の算出対象に新カラムが
   影響するか）を確認する。**content_hash への影響は Fix #06 の
   重複検出仕様と衝突しうるため必ず確認すること。**

### D. レシート読取からの変換パス

1. `/reading-receipt` の `RECEIPT_DATA` ブロック → 仕訳 JSON への変換が
   どこで行われるか（スキル指示のみか、コードが存在するか）を特定する。
2. デュアル検証の照合対象フィールド（現状 `total_amount`, `date`, `vendor`）に
   項目を追加する場合の変更箇所を列挙する。
3. レシート画像パス等の証憑参照が仕訳に記録されるか（`journals.source_file` の
   実際の使われ方）を確認する。

### E. 経過措置資料と Fix #09 の重なり

1. 報告書第10節が言及する「80%から70%、50%、30%、0%へ移行する制度資料」を
   リポジトリ内で特定する（`transitional-measures-timeline.md` ほか）。
   **既知の経過措置は 80%（〜R8.9.30）→ 50%（〜R11.9.30）→ 0% の3段階であり、
   70% / 30% という数値がどの制度・どの資料に由来するかを、
   資料のパスと該当箇所を引用して報告すること。**
   （3割特例＝売上側の特例との混同、または令和8年度改正による
   経過措置の変更のいずれかである可能性が高い。事実のみ報告し、解釈は最小限でよい）
2. 進行中の Fix #09（2026年税制改正）調査の成果物・中間報告があれば、
   その中で仕入側経過措置に触れている箇所を列挙する。

### F. 出力層の現状

1. 帳簿系コマンドの `--format csv` 実装（出力エンコーディング・改行コード・
   共通化の状況）を確認し、弥生形式ライターを追加する場合に
   再利用できる構造か、独立実装になるかを報告する。
2. CP932 でのファイル出力を行っている既存箇所の有無を確認する
   （Fix #05 は入力側の cp932 対応であり、出力側の前例があるかは未確認）。

## 4. 調査対象外

- 弥生税区分サフィックスの正確な文字列仕様（Claude 側で一次確認）
- 国税庁公表サイト Web-API / 全件データの利用規約・申請要件（Claude 側で一次確認）
- 実装方針の決定・コード変更・テスト追加（本調査の報告を受けて指示書を別途作成する)
- 少額特例等の適用要件の法的整理（制度側の整理は指示書作成時に行う）

## 5. 報告書に含める項目

1. 調査項目 A〜F の各結果。**ファイルパスと該当行・該当箇所の引用を必ず付す。**
2. 「存在しない」と結論する項目は、検索に使用したパターンと対象範囲を明記する。
3. 項目 A の結果を踏まえた、納税者属性の保持層の選択肢の列挙
   （推奨は不要、選択肢と各制約のみ）。
4. 調査中に発見した、本件と独立のバグ・不整合があれば別節で列挙する
   （修正は行わない）。

## 6. 完了条件

- 調査項目 A〜F のすべてに、引用付きの回答があること。
- 項目 E-1 について、70% / 30% の出典が資料パス付きで特定されていること。
- コードの変更が一切行われていないこと（`git status` がクリーンであること）。
